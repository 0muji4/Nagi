## 1. この文書について

本文書は、プロダクト「Nagi（凪）」が解くべき市場の課題と、それに応える設計思想・プロダクトビジョンを定義する。Nagi の文書群の最上位に位置し、各 feature の PRD/DD（最初の feature は `02_PRD_NagiTime.md` / `03_DD_NagiTime.md`）が共通の前提として参照する。

本文書が所有するのは、なぜ Nagi が要るか（市場の課題）と、Nagi が何であるか（設計思想・ビジョン・機能ロードマップ・収益方針）である。個々の feature の要件（What）と設計（How）は所有せず、各 feature の PRD/DD に委ねる。確定前の項目は【仮】と記す。

### 1.1 概要

Nagi は、効率と目的に追われる現代のスマートフォン利用に、意図的な余白を作るためのプロダクトである。ユーザーは絶え間ない通知と情報に晒され、休もうとしても無意識に操作を続けてしまい、意志だけではこの状態から抜け出せない。そこで Nagi は、何もしない時間を作り、それを操作の衝動から守ることだけに徹する。達成や継続を促す仕掛けは持たず、画面・音・触覚という穏やかな手段で、ユーザーの注意を端末から手放させる。Nagi は単一の機能ではなく、何もしない時間を守る三つのモジュールの構想からなり、最初の feature「凪の刻」がその核を担う。

## 2. 市場の課題（なぜ Nagi が要るか）

現代のスマートフォン利用は、ユーザーの注意を奪い合うアテンション・エコノミーの限界に達している。平均的な端末には約 80 個のアプリが入っているが、日常的に使われるのは約 9 個にとどまり、市場に出るアプリの大半は期待された成果に届かずに終わる。要因の一つは、機能を足し続けてインターフェースを複雑にし、ユーザーの認知負荷を高める設計にある。この環境への揺り戻しとして、要素を足すのではなく削る設計——ミニマリズムと、後述する Calm Technology / Slow Tech——への支持が広がっている。

Nagi が応える課題を、三つの「疲労」として捉える。

### 2.1 タイパ疲れ（何もしない時間が壊れる）

ユーザーは時間効率（タイパ）を追う社会的な圧力に晒され、その追求自体に疲れている。国内調査では、タイパを意識する層の約 70% が「タイパを意識しない生活を望む」と答えており、効率化は理想ではなく必要悪として受け止められている。端末の利用時間が長い層ほど、タイパへの意識は強い。

この疲労の裏返しとして、ユーザーは「何もしない時間」を欲している。しかしスマートフォンは触れれば情報を返すため、休もうとしても無意識に操作を続けてしまう。アプリを物理的にブロックするハードウェア「Brick」が支持を集めている事実は、意志だけではこの状態から抜け出せないことを多くの人が自覚し始めた証左である。凪の刻（§5）は、この無意識の操作の衝動を体験の側から押し戻すために生まれる。

### 2.2 サブスク疲労（継続課金からの離脱）

月額の継続課金（サブスク）モデルは、選択肢の多さによる決定疲れ・使っていないサービスへの支払いという財務的な負担・解約を先延ばしにする罪悪感が重なり、ユーザー離れに直面している。半年で少なくとも一つのサービスを解約する層は多数にのぼり、買い切り型やプライバシーを重視したオフラインのツールへの回帰が進む。この疲労は、Nagi の収益方針（§6）を買い切り型へ向ける根拠になる。

### 2.3 環境負荷（静かな設計への要請）

電子廃棄物の増加と、スマートフォンの製造・運用に伴う環境負荷への意識から、実行時のエネルギー消費を抑える「静かなアプリ」への要請が高まっている。視覚的なミニマリズムは、背景処理の軽量化と結びつき、電力消費とバッテリーの劣化を抑える方向と合致する。この要請は、Nagi の設計原則 Quiet Architecture（§4.3）の根拠になる。

## 3. 設計思想（どう応えるか）

これらの疲労に応える設計の核は、Amber Case が提唱する Calm Technology（穏やかな技術）にある。中心の考えは、人間の主たる務めは計算することではなく人間として生きることであり、技術は問題を解くのに必要な最小限の情報だけを差し出すべきだ、というものである。

主要な原則は次の通りである。

- 最小限の注意の要求: 技術は、ユーザーの注意の消費を可能な限り小さくする。
- 周辺的注意の活用: 情報を意識の中心ではなく周辺に置き、必要なときだけ中心へ移す。車の運転手が普段はエンジン音を意識せず、異常な音や振動のときだけ意識を向けるのと同じである。
- 語りかけない伝達: 情報を、言葉や警告ではなく、触覚や環境音の微細な変化で穏やかに届ける。
- 人間性の増幅: 機械らしい正確さをユーザーに強いるのではなく、人間の文脈を理解して操作の摩擦を減らす。

### 3.1 ミニマリズムの罠（簡素さと分かりやすさは別である）

ミニマリズムは、要素を削ること自体が目的ではない。設計者は、視覚的な簡素さと認知負荷の低さを混同しがちである。画面の美しさを追って必要な操作の手がかりを隠したり、直感的でないジェスチャーに機能を割り当てると、ユーザーの記憶への依存と操作の摩擦がかえって増える。したがって Nagi のすべての feature は、簡素さよりも先に「次の一歩が直感的に分かること」を優先する。

### 3.2 意図的な摩擦の価値

摩擦は常に悪ではない。Web で公開された「Do Nothing for 2 Minutes」は、2 分間マウスやキーボードに触れないことを求め、触れるとタイマーを最初に戻す。これは罰ではなく、休息は中断されないときに最大の効果を持つという前提に基づく仕掛けである。公開当初は訪問者の約半数がこの単純な課題に失敗したと報告されており、現代人が「操作し続けねばならない」という衝動にどれほど囚われているかを可視化した。操作を意図的に妨げて立ち止まらせること自体に価値がある——この先例が、Nagi の最初の feature（§5）の核になる。

## 4. プロダクトビジョン: Nagi

Nagi は、通知と情報の渦からユーザーを一時的に隔離し、何もしない時間を作って守る、デジタル上の聖域（サンクチュアリ）である。生産性を上げる道具ではなく、意図的な「余白」を肯定し保護することを務めとする。

主な対象は、1 日に 3 時間以上デバイスに触れ、タイパの圧力に疲れている層である。§2.1 の通り、タイパを意識する層の約 70% がそこから逃れたいと答えており、Nagi はこの層に直接応える。

Nagi のすべての feature が従う設計原則を、次に定義する。各 feature の PRD/DD は、これらを前提として要件と設計を導く。

### 4.1 Anti-Gamification（操作を促さない）

連続記録・バッジ・達成を促すプッシュ通知といった、ユーザーを画面に留めるための仕掛けを持たない。これらは「何もしない時間を守る」という務めと矛盾するためである。

### 4.2 Ambient & Tactile UI（穏やかな手段で導く）

明確な境界を持つボタンやテキストを減らし、色彩の変化・触覚・環境音でユーザーを導く。周辺的注意（§3）を活かし、ユーザーの意識の中心を占めない。端末の傾きに連動した色彩の変化なども、この穏やかな表現の一つの方向として想定する。個々の表現を各 feature で採るかどうかは、その feature の判断とする。

### 4.3 Quiet Architecture（静かな設計）

実行時のエネルギー消費を抑える設計を採る。不要な通信・過剰なログ・非効率な処理を避け、コアのロジックには、ガベージコレクションを持たない省電力な言語を含む手段を用いる。§2.3 の環境負荷への要請に応えると同時に、バッテリーの劣化を抑える。どの言語をどう用いるかという具体の選定は、各 feature の DD に委ねる。

### 4.4 Local-First（データは端末内に置く）

設定とユーザーのデータは端末内にのみ保存し、外部サーバーへ常時同期しない。これにより、開発側のサーバー維持コストをほぼゼロに抑え（§6 の収益方針を支える）、同時に「自分のデータは誰にも覗かれない」という安心をユーザーに提供する。バックアップが必要な場合は、ユーザー自身のクラウドへ暗号化してエクスポートする。

### 4.5 真に重要な割り込みは妨げない

Nagi が操作や通知を抑えるとき、電話の着信や緊急速報のような、ユーザーの安全に関わる割り込みは例外として妨げない。人間性の増幅（§3）の原則に基づき、穏やかさのためにユーザーを危険に晒さない。各 feature は、この例外を前提として設計する。

## 5. 機能ロードマップ（三つのモジュール構想）

Nagi は単一の肥大した機能ではなく、何もしない時間を守るために補い合う三つのモジュールの構想からなる。本文書は各モジュールの構想を示すにとどめ、要件（What）と設計（How）は各モジュールの PRD/DD に委ねる。

- 凪の刻（Nagi Time）— 最初の feature。ユーザーが選んだ時間だけ操作を意図的に手放すための時間を作り、進行中の無意識の操作から守る。§3.2 の意図的な摩擦を核に据える。要件は `02_PRD_NagiTime.md`、設計は `03_DD_NagiTime.md` が扱う。
- 無目的の記録（Aimless Journal）— 目的のない思考を、入力の摩擦を極小にして書き留めるための記録。別 feature の PRD/DD で扱う。
- 環境の借景（Peripheral Windows）— ホーム画面・ロック画面のウィジェットを通じて、アプリを開かなくても穏やかさを周辺的に届ける。別 feature の PRD/DD で扱う。

最初に凪の刻を作るのは、三つのうち最も「何もしない時間を守る」という務めの核に近く、§3.2 の先例によって価値が裏づけられているためである。

---

## 6. 収益方針（ビジョンレベル）

§2.2 のサブスク疲労への応答として、Nagi は月額の継続課金を避け、買い切り型を志向する。アプリへの支払いを、所有できる聖域への一度きりの投資として位置づける。§4.4 の Local-First によりサーバーの維持コストがほぼ発生しないため、買い切り型でも事業として成立する。

無料の基本機能と有料の拡張をどう分けるか、価格、環境音の拡充などの具体は、まず体験そのものの検証を優先するため最初の feature では確定させず、収益を扱う後続の検討に委ねる。

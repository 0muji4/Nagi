## 2. コンテキスト

PRD のステップと本 DD の論点は 1:1 で対応する。

| DD の論点 | 設計課題 | 対応する PRD のステップ |

## 3. 推奨設計の概要

凪の刻の体験は、描画・環境音・触覚・前面と背面の切り替えという、いずれも端末ごとの仕組みに深く結びついた要素で成り立つ。一方でこの feature には、Android で出荷した後に iOS へ展開し、その際コアを書き直さないという要請がある。深く端末に結びついた体験と、端末に依存しないコアという相反が、設計の中心問題である。

両立の鍵は、feature を「規則」と「感覚」に分けることにある。規則は、いま落ち着き・乱れ・リセット・完了のどの状態かという遷移と、その判断に用いる方針であり、端末に依存しない純粋な論理である。感覚は、その状態をユーザーに届ける画面・音・触覚であり、操作や前面背面の切り替えを受け取る入力でもあって、端末ごとの仕組みに依存する。

そこで規則を移植可能な言語 Rust のコアに、感覚を Android ネイティブの Kotlin / Jetpack Compose に置き、UniFFI が生成する境界でつなぐ。iOS へはコアを再利用し、感覚の層だけを Swift / SwiftUI で書き直せばよい。Rust を選ぶのは、ガベージコレクションを持たず実行時の負荷が小さく、Quiet Architecture に合致するためである。最初の feature では規則のコアは小さく、移植性の恩恵の多くは iOS 版で現れる。それでも小さいうちに境界を引くのは、大きくなったコードへ後付けするより、規則と感覚の絡まりを解く作業が少なく、やり直しの危険が小さいためである。

## 4. 設計論点

### Q1: 技術スタックの選定

評価軸は、移植性・低消費電力・表現の忠実さ・個人開発での保守コストの四つ。

方針は第3章の二層分離（規則 = Rust コア、感覚 = Jetpack Compose、UniFFI で接続）。移植性はコアが端末非依存ゆえ満たす。低消費電力は Rust が GC を持たないため満たす。表現の忠実さは感覚の層がネイティブゆえ Jetpack Compose・Android のハプティクス・音声 API をそのまま使えて満たす。

弱点は保守コストである。境界の構築と二言語のビルド統合が個人開発の負担になる。負担を抑えるため、コアに入れる範囲を純粋な論理だけに厳しく絞る——状態機械と引き戻しの方針（Q4）、配色の計算（Q3）、セッション記録の構造と保存（Q5）に限り、描画・音・触覚・前面背面の検知は感覚の層に残す。棄却案は §6。

### Q4: 離脱の検知・引き戻し・リセット

離脱の経路は二つあり、Android での扱いが異なる。画面上の操作は、アプリが前面にある間すべて受け取れるため、検知も引き戻しも完全にできる。アプリから出る操作は背面に回り、検知とリセットはできるが背面では演出を続けられない（PRD §7）。

そこで二経路を分けて扱う。画面上の操作には段階的に引き戻す——触れた瞬間にグラデーションを乱して警告音を微かに鳴らし、操作が続いて一定の度合いを超えたとき静かにリセットする。一度の偶発的な接触で即リセットしないのは、PRD Step 3 が求める「罰ではなく穏やかなリマインダー」を成立させるためである。アプリから出る操作には、背面への遷移を検知した時点でリセットする。背面では時間を守れない以上その時間は壊れたものとして扱い、前面復帰時は咎めず、静かに最初へ戻った状態を示す。

この引き戻しを移植可能なコアの状態機械として表す。局面は未開始・進行中・完了の三つで、落ち着きと乱れは進行中における乱れゲージ（連続量 0〜1）で表し、リセットは状態ではなく遷移として扱う。コアの入力は、開始・経過の刻み（触れているか否かを伴う）・離脱（別アプリ遷移）・割り込みの開始と終了（OS 割り込み）である。触れている間ゲージが上昇し、離すと下降して落ち着きへ戻り、グラデーションの波紋と警告音の強さをゲージに連動させる。ゲージが 1 に達するとリセットし、同じ長さの凪を最初からやり直す（経過と乱れを 0 に戻す）。離脱も即リセットとする。割り込み中は時間を進めず、終了時に再開する。初期値は上昇を毎秒 0.5（約 2 秒の連続操作で到達）・下降を毎秒 1.0（離せば約 1 秒で復帰）とし、偶発的な短い接触は許し、意図的なスクロールや短時間の繰り返しがリセットに至るように置く。これらの値はコアの方針として一箇所にまとめ、完走率（PRD §4）を見ながら調整する。【仮】リセットの戻り先（同じ凪の最初か、設定画面か）は再検討の余地がある。感覚の層は、操作とライフサイクル（前面/背面）の変化をコアへ渡し、返った状態をグラデーション・警告音・触覚で描く。

### Q3: 没入演出（グラデーション・環境音）

配色を定める計算（時間帯 → 隣り合う色相）は純粋な論理ゆえコアに置く。描画は Jetpack Compose に置き、落ち着いている間はグラデーションをごく緩やかに変化させて高頻度の再描画を避け、電力を抑える。乱れ（Q4）のときだけ波紋の動きを加える。滑らかさと低消費電力の両立は、落ち着きと乱れで描画の負荷を変えることで成り立つ。環境音は Android の音声 API（Media3 / ExoPlayer 等）で再生し、起動時にフェードイン・完了時にフェードアウトさせ（Q5）、警告音は別の層として重ねる。

### Q2: タイマー設定 UI

水滴を模した円を Jetpack Compose で描き、上下ドラッグの距離を 1〜30 分へ対応させ、指を離した時点でコアへ渡して起動する（Q4 の開始入力）。操作の手がかりを隠さない要件（`01_Background.md` §3.1 が警告する罠の回避）のため、現在選んでいる長さを表示し、触れる要素は誤操作防止に 48dp 以上（Android のタップターゲット指針）を確保する。

### Q5: 完了時の状態遷移とセッション記録

設定時間に達するとコアが完了へ遷移し（Q4）、感覚の層が環境音とグラデーションを収束させて完了を伝える。連続記録・達成の表示は置かない。セッション記録は、開始時刻・設定した長さ・結果（完走 / 中断）・中断要因の最小限に絞り、移植可能にするため構造と端末内データベースへの保存をコアに置く。記録は秘匿性の高いデータを含まないため、最初の feature では Android のアプリ専用ストレージ（端末のファイル暗号化に保護される）に任せ、独自の暗号化は加えない（本文を扱う記録 feature で暗号化の要否を改めて論じる）。

## 5. 全体設計

三層で構成する。

- **感覚の層（Jetpack Compose）**: 各画面の描画、操作とライフサイクル（前面/背面）の受け取り、グラデーション・環境音（Media3 / ExoPlayer）・触覚（VibrationEffect）。
- **規則のコア（Rust、UniFFI で公開）**: 状態機械・引き戻しの方針・配色の計算・セッション記録の構造と保存。
- **境界（UniFFI）**: 入力（開始・経過・操作・背面遷移・前面復帰・完了）と出力（状態・配色・残り時間）を受け渡す。

感覚の層が捉えた入力をコアへ渡し、コアが状態・配色・残り時間を返し、感覚の層が描く。完了またはリセット時にコアが記録を保存する。

## 6. 棄却案

- **純ネイティブ（コアも Kotlin）**: 表現と保守は最良だが、コアが Kotlin に結合し iOS で書き直しになる。PRD の前提に反し棄却。
- **KMP**: 移植性はあるが、コアが Kotlin（GC あり）になり、`01_Background.md` §4.3 が求める GC を持たない省電力な言語に反する。iOS 展開時に Kotlin/Native の実行環境も要る。棄却。
- **Flutter / React Native**: 移植性・保守は優れるが、流体的な表現・触覚・環境音の忠実さでネイティブ UI に劣り棄却。
- **背面でも演出を続ける**: リセットを避けられるが、別アプリ操作中は時間が守られておらず目的に反し、電力面でも Quiet に反し棄却。

## 7. 障害シナリオとエッジケース

- **OS 割り込み（着信・緊急速報・アラーム）**: ユーザー操作でない中断を別アプリ切替と同じにリセットすると罰と受け取られる。`01_Background.md` §4.5 が電話・緊急速報のような重要な割り込みを妨げない方針に倣い、OS 割り込みは区別して復帰時に再開する。【仮】判定方法は実装時に詰める。
- **進行中の画面ロック**: 画面を消すのは凪と矛盾しないため離脱として扱わず、タイマーを進め、環境音も鳴らし続け、ロック解除で凪の画面へ戻す。ロック中はユーザーが他アプリを操作しておらず凪に留まっているため、§6 で棄却した背面での演出継続（別アプリ操作中）とは区別する。【仮】Android で背面・ロック中に音を鳴らすには前景サービスと常駐通知が要り、Quiet・Anti-Gamification の方針との兼ね合いがある。またロックと別アプリ遷移はどちらもアプリを背面化する（画面オフのブロードキャスト等で判別する）。両者の扱いは実装時に詰める。
- **背面 / ロック中の完了の伝達**: ローカル通知は使えるが、通知を増やすのは Quiet・Anti-Gamification と引っ張り合うため、伝達手段は実装時に定める。
- **環境音と他音声の競合**: Android のオーディオフォーカス（AudioFocus）の扱いは実装時に定める。

## 8. 実装スケジュール

検証できる単位ごとに段階を分ける。

- **Phase 0 基盤**: Jetpack Compose 雛形と Rust コア＋UniFFI（Kotlin バインディング）のビルド統合で最小の状態機械を通し、Q1 の成立を最初に確かめる。
- **Phase 1 設定（Q2）→ Phase 2 没入（Q3）→ Phase 3 保護（Q4）→ Phase 4 完了と記録（Q5）→ Phase 5 調整**（完走率を基に引き戻しの強さを調整）。
- Phase 3 を最難関と見込む（Android 制約下の離脱検知が価値を左右するため）。
